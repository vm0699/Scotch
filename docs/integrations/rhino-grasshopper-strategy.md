## 2. Grasshopper Parameter Table

Each `Parameter` entry in the Scotch `ArchitectureProject` maps to a Grasshopper  
Number Slider or Panel input. The table below covers all standard parameter keys  
generated by the layout engine.

| `Parameter.key` | `label` | GH Input Type | GH Component | Range |
|---|---|---|---|---|
| `site_width` | Site Width | Number Slider | `Scotch::Site` panel | 20 – 200 ft |
| `site_depth` | Site Depth | Number Slider | `Scotch::Site` panel | 20 – 200 ft |
| `floor_height` | Floor Height | Number Slider | Building params | 8 – 14 ft |
| `{room_id}_width` | Room Width | Number Slider | Per-room cluster | min – max from server |
| `{room_id}_depth` | Room Depth | Number Slider | Per-room cluster | min – max from server |
| `orientation` | Orientation | Panel (text) | Site orientation | N / S / E / W |
| `floors` | Number of Floors | Integer Slider | Building params | 1 – 4 |
| `parking` | Parking spaces | Integer Slider | Program | 0 – 4 |
| `style` | Architectural style | Panel (text) | Program context | modern / traditional |

**How to populate sliders from JSON:**

```
Read File (path input) → GH_Parse (Human plugin or DeserializeJSON)
  → Deconstruct {parameters} list
  → Match by .key → set Slider.CurrentValue + .Minimum + .Maximum
```

With the **Human** plugin this is one "Read JSON" component + expression matching.  
Without plugins, use the GH scripting component (C# or Python) to deserialize the JSON  
and set slider values via GH_Document.FindObject.

## 3. Grasshopper Data Flow

```
scotch_project.json (File Path input)
  │
  ▼
[Read File] → raw JSON string
  │
  ▼
[Deserialize JSON]  ← Human plugin or GH Python component
  ├─ site   → Width, Depth → "Scotch::Site" surface → Extrude → Ground slab
  ├─ rooms  → data tree (branch per room: x, y, w, d, name, type)
  │     ├─ per-room Rectangle → Extrude → Wall brep
  │     └─ per-room centroid → TextDot → label
  ├─ doors  → offset + width per wall → subtracted void → BooleanDifference
  ├─ windows → offset + width + sill → void → BooleanDifference
  └─ building.floor_height → Extrude height
  │
  ▼
[Number Sliders] (one pair per room: Width / Depth)
  │  override data-tree values on change
  ▼
[Rebuild geometry] → live 3D preview
  │
  ▼
[Scotch Sync cluster] ──── POST /generate/regenerate ──→ Scotch backend
                      ←─── updated ArchitectureProject JSON ───────────
```

### Data tree branch structure

The GH definition uses a **data tree** where each branch index corresponds to one room:

```
{0} → Room 0 (x, y, width, depth, name, type)
{1} → Room 1
...
{n} → Room n
```

Grasshopper panel paths:

```
rooms[i].x     → {i}[0]
rooms[i].y     → {i}[1]
rooms[i].width → {i}[2]   ← also driven by Number Slider override
rooms[i].depth → {i}[3]   ← also driven by Number Slider override
rooms[i].name  → {i}[4]
rooms[i].type  → {i}[5]
```

## 4. Scotch Sync Cluster Specification

The "Scotch Sync" cluster (`ScotchSync.ghuser`) provides round-trip sync from GH to the  
Scotch backend, triggering regeneration and returning updated geometry.

### 4.1 Cluster inputs

| Input name | GH Type | Description |
|---|---|---|
| `JSON_Path` | File Path | Path to `scotch_project.json` |
| `Project_ID` | String | Scotch project UUID (from the file or URL) |
| `Room_Widths` | Number tree | Override widths (branch per room; null = keep original) |
| `Room_Depths` | Number tree | Override depths |
| `Sync_Button` | Boolean | Triggers sync when True |
| `API_Base` | String | Scotch API URL (default `http://localhost:8000`) |

### 4.2 Cluster outputs

| Output name | GH Type | Description |
|---|---|---|
| `Updated_JSON` | String | Raw JSON of updated `ArchitectureProject` |
| `Rooms_Tree` | Generic Data | Updated rooms data tree |
| `Warnings` | String list | Validation warnings from the server |
| `Status` | String | "OK" / error message |

### 4.3 Internal logic (GH Python component inside cluster)

```python
import json, urllib.request

def sync(api_base, project_id, room_widths, room_depths, current_json):
    project = json.loads(current_json)
    changes = []
    for i, room in enumerate(project["rooms"]):
        if room_widths[i] is not None:
            changes.append({"target_id": room["id"], "key": f"{room['id']}_width",
                            "value": room_widths[i]})
        if room_depths[i] is not None:
            changes.append({"target_id": room["id"], "key": f"{room['id']}_depth",
                            "value": room_depths[i]})
    payload = json.dumps({
        "project_id": project_id,
        "changes": changes,
        "mode": "deterministic",
    }).encode()
    req = urllib.request.Request(
        f"{api_base}/generate/regenerate",
        data=payload,
        headers={"Content-Type": "application/json"},
    )
    with urllib.request.urlopen(req, timeout=15) as resp:
        return json.loads(resp.read())["project"]
```

### 4.4 GH cluster file delivery

The `.ghuser` file is documented here rather than shipped as a binary (Grasshopper  
cluster binaries are version-specific). Users can recreate it by:

1. Opening the GH canvas and building the Python component above.
2. Grouping all Sync components → right-click → "Cluster".
3. File → Special Folders → User Object Folder → save as `ScotchSync.ghuser`.

## 6. Step-by-Step GH Workflow

1. **Export from Scotch:** Workspace → Exports → "Rhino (.py)" — runs the script and  
   saves `floor_plan_rhino.py`. Separately export JSON: "JSON" format for the GH definition.
2. **Open Grasshopper:** Rhino → Grasshopper command.
3. **File Path input:** Point to `project.json` from Scotch exports folder.
4. **Load the GH definition template** (or build from scratch using section 3 above).
5. **Adjust sliders:** each room's Width / Depth slider modifies the data tree live.
6. **Scotch Sync:** press the sync button → updated design flows back to Scotch backend.
7. **Bake:** select all geometry → right-click → Bake → Rhino model is updated.

## 7. Parameter-to-GH Reference

Full mapping from every `Parameter` the generator emits to a GH equivalent:

```
project.site.width                → Number Slider "Site W"    range [20, 200]  step 1
project.site.depth                → Number Slider "Site D"    range [20, 200]  step 1
project.building.floor_height     → Number Slider "Fl. Ht."   range [8, 14]    step 0.5
project.building.floors           → Integer Slider "Floors"   range [1, 4]
rooms[i].width                    → Number Slider "Rm{i} W"   range [param.min, param.max] step 0.5
rooms[i].depth                    → Number Slider "Rm{i} D"   range [param.min, param.max] step 0.5
project.site.orientation          → Panel "Orientation"        N/S/E/W
```

The `Parameter.min` and `Parameter.max` fields from the Scotch API should be used to  
set slider bounds — they encode the valid range derived from the room type defaults library.

## 8. Known Limitations

- **Boolean reliability in Rhino:** rooms with very narrow walls (< 4 in) or tight  
  tolerances may need a manual `BooleanDifference` pass after import.
- **GH live sync requires Scotch backend:** `http://localhost:8000` must be running.  
  Cloud sync is planned for Phase 18.
- **Grasshopper slider count:** a large project (> 8 rooms) produces many sliders;  
  use panel groups or Pufferfish data-tree operations to keep the canvas manageable.
- **RhinoPython + CPython 3:** Rhino 8 ships CPython 3 by default. The generated script  
  uses `rhinoscriptsyntax` which works in both IronPython 2.7 (Rhino 7) and CPython 3 (Rhino 8).
