# Cloud Storage Strategy — Scotch (Phase 18.3)

Maps every local filesystem path to its cloud equivalent so the
`CloudProjectStore` implementation (Phase 18+) can be written with a
clear object-key contract.

---

## Local → cloud path mapping

| Local path | Cloud object key | Notes |
|---|---|---|
| `data/users/{uid}/projects/{pid}/project.json` | `users/{uid}/projects/{pid}/project.json` | Full `StoredProject` envelope |
| `data/users/{uid}/projects/{pid}/exports/manifest.json` | `users/{uid}/projects/{pid}/exports/manifest.json` | Export manifest list |
| `data/users/{uid}/projects/{pid}/exports/{filename}` | `users/{uid}/projects/{pid}/exports/{filename}` | Exported file |

The `data/` prefix is stripped; everything else is identical. This means
the same `ProjectStore` interface works in both environments.

---

## Storage provider options

### Option A: AWS S3

```python
# CloudProjectStore.get_project()
obj = s3.get_object(
    Bucket=self.bucket,
    Key=f"users/{user_id}/projects/{project_id}/project.json"
)
return StoredProject.model_validate_json(obj["Body"].read())
```

- Use `s3.put_object` with `ServerSideEncryption="AES256"` for all writes.
- Versioning on the bucket provides implicit history before Phase 19 lands.
- Use S3 Intelligent-Tiering for export files (rarely accessed after download).

### Option B: Supabase Storage (recommended for Phase 18)

Supabase offers S3-compatible object storage with built-in auth integration.
The `user_id` from the JWT (see `auth-strategy.md`) maps directly to a
Supabase RLS policy on the bucket.

```sql
-- Supabase Storage policy
CREATE POLICY "Users own their files"
ON storage.objects FOR ALL
USING (auth.uid()::text = (storage.foldername(name))[2]);
-- path: users/{uid}/projects/...
--                   ^^ position 2 of the path
```

## Signed URL exports

Export files can be large. Instead of streaming through the API, the
download endpoint generates a signed URL valid for 15 minutes:

```python
# CloudProjectStore.get_export_path() returns a signed URL wrapper
url = s3.generate_presigned_url(
    "get_object",
    Params={"Bucket": bucket, "Key": key},
    ExpiresIn=900,
)
# Frontend downloads directly from S3 — no API bandwidth cost
```

The `ProjectStore.get_export_path()` interface returns a `Path` today.
For cloud, it would return a `CloudPath` wrapper (or the route could call
a separate `get_export_url()` method added in Phase 18).

## Atomic writes

Local writes are atomic via `os.replace` (temp file + rename). Cloud writes
must use:
- **S3:** `put_object` is atomic per object (no partial-read issue).
- **Supabase:** `upsert` in the Storage API is idempotent.

For `StoredProject` updates, always write the full JSON (no patch), so a
partial-write failure leaves the previous version intact (S3 keeps the old
version if versioning is enabled).

## Multi-region considerations

| Concern | Approach |
|---|---|
| Latency | Deploy in the same region as the Postgres instance |
| Compliance (GDPR) | Use EU region (`eu-west-1` / Supabase EU) for EU customers |
| CDN for exports | CloudFront / Supabase CDN for large export files |
| Backup | S3 Cross-Region Replication or Supabase daily backups |
